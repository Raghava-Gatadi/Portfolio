Digital Forensics: AWS Breach Timeline Reconstruction

By Raghava Gatadi • September 5, 2024

Conducted a comprehensive digital forensics investigation to reconstruct a 48-minute AWS infrastructure breach. Analyzed raw CloudTrail logs to identify attack vectors, tactics, and impact scope.

Initial Compromise

Detected initial access via a compromised API key from an external IP address located in a known malicious region. The attacker began with reconnaissance activities, enumerating IAM roles, EC2 instances, and S3 buckets.

Log Analysis Methodology

Parsed raw CloudTrail logs using jq and regex to reconstruct the attack timeline. Applied filtering techniques to isolate anomalous activity patterns and correlate events across multiple AWS services.

Attack Progression

The breach followed a classic kill chain pattern:

  1. Reconnaissance: 12 minutes of API calls to enumerate resources
  2. Credential Access: 8 minutes attempting to access IAM policies and roles
  3. Discovery: 15 minutes mapping internal network topology
  4. Exfiltration: 10 minutes downloading data from S3 buckets
  5. Impact: 3 minutes modifying security groups and deleting logs

Timeline Reconstruction

Created a granular timeline showing the exact sequence of events with timestamps accurate to the second. Identified the specific IAM permissions that enabled lateral movement and privilege escalation.

Evidence Preservation

Implemented proper evidence handling procedures, creating forensic copies of relevant log files and maintaining chain of custody documentation. Preserved artifacts for potential legal proceedings.

Root Cause Analysis

Determined the breach originated from a developer's API key that was inadvertently exposed in a public GitHub repository. The key had excessive permissions violating the principle of least privilege.

Key Takeaways:

  • Proper log retention and analysis capabilities are essential for incident response
  • Implementing least-privilege access policies limits blast radius during breaches
  • Automated monitoring for exposed credentials in public repositories is critical
  • Forensic readiness procedures should be established before incidents occur
